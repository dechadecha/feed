# Развёртывание на своём сервере

Инструкция для того, кто ставит генератор фида у себя. Всё берётся из репозитория,
ставится одной командой.

```
РЕПОЗИТОРИЙ:  https://github.com/dechadecha/feed.git
```

---

## Установка

```bash
git clone https://github.com/dechadecha/feed.git feed
cd feed
sudo bash setup.sh
```

Всё. Через пару минут скрипт напечатает готовую ссылку для кабинета Директа.

Что он делает сам:

- определяет публичный адрес машины;
- проверяет python и что сервер видит API клиента;
- ставит nginx, если его нет;
- раскладывает файлы в `/opt/feed` и `/var/www/html/feeds`;
- заводит systemd-таймер на ежедневный запуск (или cron, если systemd нет);
- **собирает фид сразу**, не дожидаясь утра;
- скачивает картинки и настраивает их раздачу со своего сервера;
- проверяет результат и печатает ссылку.

Если автоопределение адреса промахнулось или нужен домен — задать явно:

```bash
sudo bash setup.sh 203.0.113.10
sudo bash setup.sh feeds.example.com
```

Повторный запуск безопасен: скрипт идемпотентный, просто перенастраивает поверх.

## Что понадобится

- Сервер с Ubuntu/Debian и **белым IP** или доменом — Директ должен дотянуться
  до файла снаружи. Годится любой VPS.
- **Python 3.8+** — обычно уже стоит. Никаких пакетов ставить не надо.
- Доступ в кабинет Яндекс Директа, чтобы подключить фид.
- **Три параметра источника данных** — адрес API, адрес сайта и слаг проекта.
  Их выдаёт владелец репозитория отдельно, в публичной инструкции их нет.
  Скрипт спросит их при первом запуске и сохранит в `/opt/feed/feed.conf`;
  при повторных запусках берёт оттуда. Неинтерактивно:
  `FEED_API=... FEED_SITE=... FEED_PROJECT=... sudo -E bash setup.sh`

Сервер должен видеть сайт-источник — скрипт это проверит и скажет, если не выходит.

## Дальше

Фид обновляется сам каждое утро в 05:20 МСК. Ничего делать не нужно.

```bash
sudo systemctl start feed           # пересобрать прямо сейчас
sudo journalctl -u feed -n 20       # лог последних прогонов
systemctl list-timers feed.timer    # когда следующий запуск
```

`systemctl is-active feed.service` показывает `inactive` — это норма для
разовых задач. Смотреть надо `systemctl show feed.service -p Result`,
там должно быть `success`.

## Обновление кода

```bash
cd feed && git pull
sudo bash setup.sh
```

## Если что-то пошло не так

| Симптом | Причина |
|---------|---------|
| «API отвечает не 200» | сервер не видит сайт-источник — firewall, DNS, гео-блокировка |
| «не смог определить адрес» | задать вручную: `sudo bash setup.sh <IP>` |
| Фид не открывается снаружи | закрыт 80 порт или nginx раздаёт не `/var/www/html` |
| Директ не принимает файл | выбран тип «Недвижимость» вместо «Другой бизнес» / YML |
| Директ: «нет страниц каталога» | в кабинет попала старая версия фида, обновить |
| В превью серые квадраты | Директ ещё не забрал картинки, подождать несколько часов |
| Прогон падает с `SystemExit` | API вернул пустой список; фид намеренно не перезаписан |
| `ModuleNotFoundError` | в код добавили стороннюю библиотеку — так делать нельзя |

## HTTPS

Не обязателен, Директ принимает и http. Если нужен — понадобится домен,
направленный A-записью на IP сервера:

```bash
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d <ваш-домен>
sudo bash setup.sh <ваш-домен>     # пересобрать фид со ссылками на https
```

Второй шаг обязателен: внутри фида лежат ссылки на картинки, их тоже надо
перевести на новый адрес.

## Про доступы

В проекте нет секретов: ни ключей, ни токенов, ни паролей. Генератор читает
открытый API сайта-источника, который работает без авторизации. Сами адреса
источника в репозитории не публикуются — их выдаёт владелец (см. выше).

Поэтому для установки не нужны ничьи чужие доступы — ни к серверу автора,
ни к его кабинету Директа. Всё разворачивается на своей машине под своим ключом.

Единственное, что не стоит коммитить, — готовые выгрузки фида (`*.xml`).
Они не секретны, но мгновенно устаревают и путают; в `.gitignore` уже учтены.

## Как это устроено внутри

Коротко, для понимания. Подробности — в `HOW_IT_WORKS.md` и `CLAUDE.md`.

- **Источник.** Три режима (`FEED_SOURCE`, установщик спросит при первом запуске):
  `api` — открытый JSON API сайта; `yrl` — готовый YRL-фид застройщика (выгрузка
  для Циана / Я.Недвижимости) с конвертацией в YML; `csv` — таблица от клиента.
  Новый тип источника добавляется одной функцией в build_feed.py.

### Источник csv — какие колонки нужны в таблице

Колонки ищутся по названиям (регистр и лишние пробелы неважны, синонимов много).
**Обязательные** — без них строка пропускается: `id` (Код лота / Артикул),
`Ссылка` (или задать `FEED_URL_TEMPLATE` с `{id}`, если ссылок нет),
`Комнатность` (0 или «студия»), `Площадь`, `Цена`.
**Необязательные** — подтянутся, если есть: `Старая цена` (даёт шильдик скидки),
`Этаж`, `Этажность`, `Корпус`, `Секция`, `Номер квартиры`, `Цена за метр`,
`Отделка`, `Удобства` (через запятую), `Акция`, `Планировка` (ссылка на картинку),
`Срок сдачи`, `Адрес`.

Цену можно писать как угодно («9 500 000 ₽»), площадь — с запятой («42,5»).
Если обязательной колонки нет, установщик/генератор напечатает список
распознаваемых названий. Google Sheets — «Файл → Опубликовать → CSV», ссылку
в `FEED_CSV_URL`. Локальный файл — в `FEED_CSV_PATH`.
- **Данные.** Открытый REST API: список лотов, карточка каждого, карточка ЖК.
  Раз в сутки, последовательно, с паузой 0.5 с — чтобы не создавать всплеск
  нагрузки на чужой сервер.
- **Формат.** YML, а не YRL: только в нём есть `<oldprice>`, а значит шильдик
  со скидкой и две цены в объявлении.
- **Картинки.** Планировки лежат как PNG с прозрачным фоном и разными пропорциями.
  Они прогоняются через imgproxy сайта клиента (поля, белый фон, JPG), результат
  скачивается **один раз** и дальше раздаётся с вашего сервера. Директ ходит
  только к вам.
- **Надёжность.** XML пишется атомарно; при пустом ответе API старый фид не
  затирается; если картинку не удалось скачать — остаётся вчерашняя, а ссылка
  не меняется, поэтому Директ не перекачивает всё заново.
